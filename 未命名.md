操作系统的核心职能之一就是管理计算机硬件资源。为了高效、安全地支持多道程序并发执行，操作系统主要采用了**复用（Multiplexing）**、**虚拟（Virtualization）** 和**抽象（Abstraction）** 这三类核心技术。

以下是对这三类技术的详细描述及举例：

### 1. 复用技术 (Multiplexing)

**描述：** 复用是指让多个进程或用户“共享”同一个物理资源。由于物理资源通常是有限的（如只有一个CPU、一块网卡），操作系统通过特定的调度策略，让多个请求者在逻辑上同时使用该资源。复用主要分为两种形式：

- **时分复用 (Time-Division Multiplexing)：** 将资源的使用权按**时间片**轮流分配给各个进程。从宏观上看，多个进程似乎在同时运行；但从微观上看，它们在交替使用资源。这种方式要求资源必须是可抢占的或能够快速切换上下文的。
- **空分复用 (Space-Division Multiplexing)：** 将资源在**空间**上划分成若干部分，分配给不同的进程同时使用。每个进程占用资源的一个独立子集，彼此互不干扰。

**举例：**

- **时分复用：**
    - **CPU：** 操作系统通过进程调度算法（如时间片轮转），让CPU在毫秒级的时间粒度上快速切换执行不同的进程。
    - **网络带宽：** 在TCP/IP协议栈中，多个应用程序的数据包被交织发送到同一根网线上。
- **空分复用：**
    - **内存：** 操作系统将物理内存划分为不同的分区或页框，进程A占用0-1GB区域，进程B占用1-2GB区域，两者同时驻留在内存中。
    - **磁盘：** 文件系统将磁盘空间划分为不同的块或扇区，不同文件存储在不同的物理位置上。
    - **GPU显存：** 现代GPU支持多实例技术，将显存和计算单元按比例切分给不同的容器或虚拟机使用。

---

### 2. 虚拟化技术 (Virtualization)

**描述：** 虚拟化是将一个物理资源变换为多个逻辑上的对应物，或者将多个物理资源整合为一个逻辑资源。其核心目的是**解耦**：让用户/程序感觉拥有专属的、无限的或标准化的资源，而无需关心底层物理资源的实际数量、位置或异构性。虚拟化通常建立在复用技术之上，但更强调“幻觉”的构建和隔离性。

**与复用的区别：** 复用侧重于“如何分”，虚拟化侧重于“看起来是什么”。例如，内存的分页是复用（空分），但让每个进程都以为自己拥有从0开始的连续4GB地址空间，则是虚拟化。

**举例：**

- **虚拟内存：** 这是最经典的例子。物理内存可能只有8GB，但操作系统通过页表和磁盘交换区，让每个进程都认为自己独享了巨大的、连续的虚拟地址空间。
- **虚拟机 (VM)：** 通过Hypervisor（如KVM, VMware），在一台物理服务器上模拟出多台完整的计算机，每台都有独立的虚拟CPU、内存和设备。
- **虚拟文件系统 (VFS)：** Linux的VFS层将Ext4、NFS、NTFS等不同格式的存储设备统一抽象为标准的文件操作接口，对用户而言，所有存储都是统一的“文件树”。
- **RAID：** 将多块物理磁盘虚拟化为一块逻辑磁盘，提供更大的容量或冗余保护。

---

### 3. 抽象技术 (Abstraction)

**描述：** 抽象是通过定义良好的接口（API/系统调用）隐藏资源的复杂实现细节，向用户提供简化、统一的操作模型。抽象是操作系统作为“扩展机器”功能的体现，它降低了编程难度，提高了软件的可移植性和安全性。

**核心价值：** “机制与策略分离”以及“屏蔽差异”。用户只需要知道“做什么”（What），而不需要知道“怎么做”（How）。

**举例：**

- **进程 (Process)：** 这是对CPU执行流的抽象。程序员不需要了解寄存器保存、上下文切换、TLB刷新等底层细节，只需调用 `fork()` 和 `exec()` 即可创建和管理执行单元。
- **文件 (File)：** 这是对持久化存储设备的抽象。无论底层是SSD、HDD还是网络存储，也无论数据是如何分块、索引、纠错的，程序员只需使用 `read()/write()` 这种字节流模型进行操作。
- **Socket：** 这是对网络通信的抽象。开发者无需处理数据包组装、路由选择、拥塞控制等复杂协议细节，只需像读写文件一样进行网络数据传输。
- **设备驱动接口：** 操作系统定义了标准的块设备/字符设备接口，使得上层应用可以用统一的方式访问键盘、鼠标、打印机等千差万别的硬件。

### 总结对比

|技术|核心问题|关键词|典型代表|
|:--|:--|:--|:--|
|**复用**|资源不够分怎么办？|共享、时间片、分区|CPU调度、内存分区|
|**虚拟**|如何让资源更好用/更多？|映射、解耦、幻觉|虚拟内存、虚拟机|
|**抽象**|如何让资源更易用？|接口、封装、隐藏细节|进程、文件、Socket|

这三者并非孤立存在，而是层层递进、相互支撑的：**抽象**定义了资源的逻辑模型，**虚拟化**扩展了资源的逻辑边界，而**复用**则是实现前两者的底层物理基础。



# 模式切换 VS 进程切换

> **模式切换是进程切换的必要前提（子集），但绝大多数模式切换并不会引发进程切换。** 
> 进程切换一定包含模式切换，但模式切换==远不等于==进程切换。

## 1. 本质区别
| 维度   | 模式切换 (Mode Switch)              | 进程切换 (Process/Context Switch) |
| :--- | :------------------------------ | :---------------------------- |
| 切换什么 | CPU 的特权级（用户态 ↔ 内核态）             | CPU 的执行流（进程A ↔ 进程B）           |
| 核心操作 | 修改状态寄存器（如 x86 CS/EFLAGS）、跳转入口地址 | 保存/恢复完整上下文（寄存器、PCB、页表、TLB等）   |
| 所属主体 | 同一个进程内部的状态变化                    | 两个不同进程之间的控制权转移                |
| 类比   | 同一个人换上工作服进入车间                   | 换了一个全新的人来操作机器                 |
## 2. 开销量级的巨大差异

这是两者最实际的区别，直接决定了系统性能：
- **模式切换：轻量级（~几百纳秒）**
    - 仅涉及少量寄存器的硬件自动保存/恢复
    - **不刷新 TLB**（因为还是同一个进程的地址空间）
    - **不切换页表基址寄存器**（CR3/TTBR0 不变）
    - Cache 命中率几乎不受影响
- **进程切换：重量级（~几微秒到几十微秒）**
    - 保存/恢复所有通用寄存器、浮点/SIMD 寄存器
    - **切换页表** → 可能导致 **TLB 全量刷新**（除非有 PCID/ASID 支持）
    - Cache 被新进程的数据污染，旧进程的缓存行失效
    - 可能触发流水线冲刷、分支预测器重置
    - 涉及 PCB 读写、调度器决策、runqueue 锁竞争

> ⚡️ **数量级对比：** 一次进程切换的开销通常是模式切换的 **10~100倍**。这就是为什么现代 OS 极力减少不必要的进程切换，而模式切换（如系统调用）则通过 `syscall/sysret` 等快速路径优化到极致。

## 3. 进程切换和模式切换的具体场景
```
├── ✅ 不导致进程切换（占绝大多数！） 
│ ├── 系统调用返回用户态（read/write 完成后回到原进程） 
│ ├── 中断处理完毕返回被中断的进程（时钟中断后继续跑原进程） 
│ ├── 异常/缺页处理完毕返回原进程 
│ └── 信号处理函数执行完毕返回用户态 
│ 
└── ❌ 导致进程切换（仅少数情况） 
	├── 时间片耗尽（时钟中断 → schedule() → 切换到新进程） 
	├── I/O阻塞（read 无数据 → sleep → schedule() → 切换） 
	├── 主动让出CPU（yield/sleep/mutex_lock 阻塞） 
	├── 高优先级进程就绪（抢占式调度） └── 进程终止（exit → schedule() → 切换）
```

### 4. 进程切换中模式切换的角色

> 当进程切换确实发生时，模式切换在其中扮演什么角色？

```
进程A (用户态) 
    ↓ [模式切换①: 用户→内核] ← 触发源(中断/系统调用)
内核: 保存A的上下文到PCB_A
内核: 调度器选择进程B
内核: 从PCB_B恢复B的上下文
    ↓ [模式切换②: 内核→用户] ← iret/eret 指令
进程B (用户态)
```

- **模式切换①** 是进程切换的**入口**：没有它，内核无法获得控制权来做切换决策。
- **模式切换②** 是进程切换的**出口**：切换完成后必须回到用户态才能执行新进程的用户代码。
- **中间的"保存/恢复/调度"** 才是进程切换的真正内容，这部分全程在内核态完成，不涉及模式切换。

### 5. 一个特殊的边界情况：内核线程

内核线程（如 Linux 的 `ksoftirqd`, `migration`）**只运行在内核态**，它们之间的切换：
- ✅ 是进程切换（不同执行流、不同PCB、可能不同页表）
- ❌ **不包含模式切换**（始终在内核态，无需切换特权级）

这进一步证明：**进程切换和模式切换是两个独立的概念，只是在大多数场景下恰好同时发生。**

### 总结对照表

| 对比项            | 模式切换          | 进程切换             |
| -------------- | ------------- | ---------------- |
| **是否改变当前进程**   | ❌ 否           | ✅ 是              |
| **是否改变特权级**    | ✅ 是           | 不一定（内核线程间切换就不变）  |
| **是否切换页表/TLB** | ❌ 否           | ✅ 通常是            |
| **开销**         | 极小（ns级）       | 较大（μs级）          |
| **频率**         | 极高            | 相对较低             |
| **必要性**        | 进程切换通常需要它作为桥梁 | 模式切换不需要它         |
| **硬件参与程度**     | 主要由硬件自动完成     | 主要由软件（OS）完成，硬件辅助 |
> 模式切换是**CPU特权级**的变更，进程切换是**执行流**的变更。进程切换通常以**两次模式切换**为边界（进内核+出内核），但绝大多数模式切换（如系统调用返回、中断返回）并不伴随进程切换。两者的**开销相差1-2个数量级**，区分它们是理解OS性能优化的基础。
